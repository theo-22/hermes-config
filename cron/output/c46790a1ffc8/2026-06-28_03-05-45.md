# Meta-Agent Pattern

Build agents that maintain other agents. A meta-agent reads from multiple data sources, compares actual activity against documented state, and updates project contexts automatically — so humans and AI sessions start with fresh context.

## When to Use

Trigger when:
- Project Rooms are drifting from actual activity
- You need to check multiple sources (Pieces, sessions, files, cron outputs) for project health
- You want to automate the "what changed since last session" workflow
- You're building a system that maintains other agents or project contexts

## Architecture

```
Data Sources → Meta-Agent → Project Room Updates
     ↓              ↓              ↓
Pieces digest   Compare      CURRENT_STATE.md
Session logs    actual vs     NEXT_ACTION.md
File changes    documented    Meta-report
Cron outputs    state
```

### Data Sources (in priority order)
1. **Pieces daily digest** — `~/Hermes/Working/pieces_continuity_YYYY-MM-DD.md`
2. **Session transcripts** — via `session_search` tool
3. **File changes** — git diff or mtime checks in Project Room directories
4. **Cron job outputs** — evaluation reports, review files

### Outputs
- Updated `CURRENT_STATE.md` and `NEXT_ACTION.md` per Project Room
- Daily meta-report at `~/Operations/reports/meta_agent_YYYY-MM-DD.md`
- Action log at `~/Operations/state/meta_agent_log.jsonl`

## Implementation Pattern

### 1. Script Structure
```python
# meta_agent_sweep.py
def main():
    # 1. Gather data sources
    pieces_digest = get_pieces_digest()
    sessions = get_recent_sessions(hours=24)
    
    # 2. Analyze each Project Room
    for room in get_project_rooms():
        analysis = analyze_project_room(room)
        update_project_room(room, analysis)
    
    # 3. Generate report
    report = generate_report(analyses, pieces_digest, sessions)
    write_report(report)
```

### 2. Staleness Detection
- **Active**: Updated within 14 days (and lifecycle is "Active")
- **Review**: No update in 14-30 days (and lifecycle is "Active") — flag for human review
- **Stale**: No update in 30+ days (and lifecycle is "Active") — mark as stale
- **Maintenance rooms**: NEVER flagged for staleness — they are quiet by design

### 3. Update Rules (Principles, Not Hard Rules)
- Maintenance rooms are quiet by design — never flag for staleness
- If room hasn't been touched in 30+ days AND lifecycle is "Active" AND no live strand exists → mark stale
- If git shows recent activity but state file is old → the room may still be active; check content before changing
- Never auto-archive — that requires human decision
- Always record reasoning in the action log

### 4. Cron Schedule
Run during cheap hours (2 AM) to avoid competition with other work:
```
0 2 * * * meta_agent_sweep.py
```

## Key Principles (from Nate B Jones video — 2026-06-16)

1. **Architecture over tools** — Works with any source, not tied to one tool
2. **Principles over rules** — "If stale and no live strand, mark stale" not hard-coded thresholds
3. **Agent builds = agent maintains** — Meta-agent records reasoning so future runs understand past decisions
4. **Infrastructure, not tool** — Other agents (Claude, Codex, Hermes) can read the updated Project Rooms

## Report-Only by Default (Codex refinement)

The meta-agent sweep runs report-only by default. It does NOT update Project Room files unless `--write-check-stamp` is explicitly passed. This prevents making rooms look fresh just by checking them.

## Content-Aware Analysis (Codex refinement)

The meta-agent reads actual file content, not just file mtime:
- `Room state last updated:` field
- `Card checked:` field
- `Lifecycle state:` field
- `Status:` field
- `Blocked:` field

This is more accurate than mtime alone.

## Pitfalls

- **Don't auto-archive** — marking stale is fine, archiving requires human decision
- **Don't over-update** — if state file is recent, don't touch it
- **Record reasoning** — every update should have a log entry explaining why
- **Respect room-local authority** — if CURRENT_STATE.md says "Active" but mtime is old, check git before changing
- **Handle errors gracefully** — if Pieces digest is missing, skip that source; don't fail the whole sweep
- **Don't use Pieces QGPT API** — it's broken on PiecesOS 12.4.1 (500 errors). Use read-only tools or file-based relay.
- **Cron scripts must be in ~/.hermes/scripts/** — use symlinks for scripts in Operations/scripts/
- **computer_use not available in chat** — only works in TUI/desktop sessions
