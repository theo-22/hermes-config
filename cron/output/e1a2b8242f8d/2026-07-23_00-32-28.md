# Meta-Agent Pattern

Build agents that maintain other agents. A meta-agent reads from multiple data sources, compares actual activity against documented state, and updates project contexts automatically — so humans and AI sessions start with fresh context.

## When to Use

Trigger when:
- Project Rooms are drifting from actual activity
- You need to check multiple sources (Pieces, sessions, files, cron outputs) for project health
- You want to automate the "what changed since last session" workflow
- You're building a system that maintains other agents or project contexts

## Architecture

```
Data Sources → Meta-Agent → Project Room Updates
     ↓              ↓              ↓
Pieces digest   Compare      CURRENT_STATE.md
Session logs    actual vs     NEXT_ACTION.md
File changes    documented    Meta-report
Cron outputs    state
```

### Data Sources (in priority order)
1. **Pieces daily digest** — `~/Hermes/Working/pieces_continuity_YYYY-MM-DD.md`
2. **Session transcripts** — via `session_search` tool
3. **File changes** — git diff or mtime checks in Project Room directories
4. **Cron job outputs** — evaluation reports, review files

### Outputs
- Updated `CURRENT_STATE.md` and `NEXT_ACTION.md` per Project Room
- Daily meta-report at `~/Operations/reports/meta_agent_YYYY-MM-DD.md`
- Action log at `~/Operations/state/meta_agent_log.jsonl`

## Implementation Pattern

### 1. Script Structure
```python
# meta_agent_sweep.py
def main():
    # 1. Gather data sources
    pieces_digest = get_pieces_digest()
    sessions = get_recent_sessions(hours=24)
    
    # 2. Analyze each Project Room
    for room in get_project_rooms():
        analysis = analyze_project_room(room)
        update_project_room(room, analysis)
    
    # 3. Generate report
    report = generate_report(analyses, pieces_digest, sessions)
    write_report(report)
```

### 1a. Room discovery — canonical Home directory scan

The sweep discovers rooms by scanning five canonical Home directories for subdirectories that contain `CURRENT_STATE.md`:

```python
HOME_DIRS = [
    Path("/Volumes/Extra/Substrate/Commons"),
    Path("/Volumes/Extra/Substrate/Roles"),
    Path("/Volumes/Extra/Substrate/Hermes"),
    Path("/Volumes/Extra/Substrate/Brain"),
    Path("/Volumes/Extra/Substrate/InterfaceFlow"),
]

def get_project_rooms() -> list[Path]:
    rooms = []
    for home in HOME_DIRS:
        if not home.exists():
            continue
        for item in home.iterdir():
            if item.is_dir() and (item / "CURRENT_STATE.md").exists() and item.name not in ROOM_EXCLUDE:
                rooms.append(item)
    return sorted(rooms)
```

This replaced the old `~/Projects/` symlink scan (2026-07-11). Key changes from the old approach:
- **Discovery signal:** `CURRENT_STATE.md` presence (was `README.md` presence)
- **Search scope:** 5 canonical Home directories (was a single `~/Projects/` directory)
- **Excludes** generic names like `room` (staging artifact) that happen to have CURRENT_STATE.md
- **Symlinks no longer required:** rooms that never had a `~/Projects/` symlink (e.g., `model_testbench`) are automatically discovered

**ROOM_EXCLUDE:** set `{"room"}` to prevent staging artifacts (e.g., `InterfaceFlow/room/` — a pre-cutover staged room model) from being counted as Project Rooms. Add to this set if other non-room directories with CURRENT_STATE.md appear.

**Before migrating to Home scan:** The sweep used `BASE / "Projects"` which resolved through `~/Projects/` compatibility symlinks. That approach missed rooms that lacked symlinks and included staging artifacts. The canonical scan fixed both.

### 2. Staleness Detection
- **Active**: Updated within 14 days (and lifecycle is "Active")
- **Review**: No update in 14-30 days (and lifecycle is "Active") — flag for human review
- **Stale**: No update in 30+ days (and lifecycle is "Active") — mark as stale
- **Maintenance rooms**: NEVER flagged for staleness — they are quiet by design

### 3. Update Rules (Principles, Not Hard Rules)
- Maintenance rooms are quiet by design — never flag for staleness
- If room hasn't been touched in 30+ days AND lifecycle is "Active" AND no live strand exists → mark stale
- If git shows recent activity but state file is old → the room may still be active; check content before changing
- Never auto-archive — that requires human decision
- Always record reasoning in the action log

### 4. Cron Schedule

Run outside DeepSeek peak windows (W1: 9pm-midnight, W2: 2am-6am EDT — both 2x pricing). Safe window: 6am-9pm EDT. No-agent scripts ($0) can run any time:
```
30 0 * * * meta_agent_sweep.py    # 12:30am — between W1 and W2, off-peak
```

## Report-Only by Default (Codex refinement)

The meta-agent sweep runs report-only by default. It does NOT update Project Room files unless `--write-check-stamp` is explicitly passed. This prevents making rooms look fresh just by checking them.

## Content-Aware Analysis (Codex refinement)

The meta-agent reads actual file content, not just file mtime:
- `Room state last updated:` field
- `Card checked:` field
- `Lifecycle state:` field
- `Status:` field
- `Blocked:` field

This is more accurate than mtime alone.

## Stale-Flag Triage

When the sweep report flags rooms as "Needs Action" or "Worth A Quick Look", not every flagged room needs a lifecycle change. Determine the right response:

1. Read the room's `CURRENT_STATE.md` — check the Lifecycle state and the `Card checked:` / `Room state last updated:` dates
2. **"Quiet until trigger" rooms** — intentionally dormant, but the meta-agent **will re-flag them** at the 30-day threshold regardless of lifecycle. If they're >30 days old, they need a **date refresh** (not a lifecycle change). See "Quiet Room Refresh Pass" below.
3. **"Active" rooms waiting on Ted routing** are not stale — no action unless the "Card checked" date is outdated
   - **Exception: when Ted explicitly routes a cleanup pass** and meta-agent flagged "Worth A Quick Look" rooms have `Card checked` >7 days stale, bump the date. Do NOT change lifecycle or content — just `Card checked:`, `Room state last updated:`, meta-agent check stamp, and the dashboard row. This prevents them from re-flagging on the next sweep. (Real case 2026-07-08: Project_Interface_Workspace_Flow and Research_Coordination, both Active at 10 days, bumped by Ted's route.)
4. **"Settled / historical" rooms** — already handled; verify the dashboard row agrees
5. **Only change lifecycle** when the room's lifecycle state in CURRENT_STATE.md disagrees with its row in `PROJECT_ROOMS_STATUS.md`

See `references/stale-flag-triage-pattern-2026-07-04.md` for the full decision tree and a worked example.

## Pitfalls

- **Don't auto-archive** — marking stale is fine, archiving requires human decision
- **Don't over-update** — if state file is recent, don't touch it
- **Record reasoning** — every update should have a log entry explaining why
- **Respect room-local authority** — if CURRENT_STATE.md says "Active" but mtime is old, check git before changing
- **Handle errors gracefully** — if Pieces digest is missing, skip that source; don't fail the whole sweep
- **Don't use Pieces QGPT API** — it's broken on PiecesOS 12.4.1 (500 errors). Use read-only tools or file-based relay.
- **Script sync after any actor edits a cron script:** The source `~/.hermes/scripts/` and profile runtime `~/.hermes/profiles/<profile>/scripts/` can diverge. After any actor edits a cron script, compare both copies. The profile dir is what cron actually runs — source changes alone don't take effect. Check both when debugging cron errors. Known real case (2026-07-04): Claude edited meta_agent_sweep.py source to a 193-byte wrapper but the profile still ran the old 22KB version, and 4 scripts were missing from the profile entirely.
- **computer_use not available in chat** — only works in TUI/desktop sessions
- **Symlinks from ~/.hermes/scripts/ to ~/Operations/scripts/ ARE accepted** — the cron gateway follows symlinks. This was relaxed; earlier versions may have rejected them.
- **Deliver setting:** Use `deliver: "origin"` not `deliver: "local"` — "local" causes false errors on any stdout output.
- **Script exit codes:** Always return 0 on successful completion. Encode status in the output file, not the exit code (YELLOW=1, RED=2 were flagged as errors).
- **Recreating deleted cron jobs:** If a cron job's script was deleted in a reorganization, check the backup at `/Users/ted/Desktop/hermes-merge-backup-20260612-094507/live-home/scripts/` for the original, then recreate the cron job with the restored script.
- **Script location for profile cron jobs:** Cron jobs in a profile (e.g., substrate-hermes) look for scripts in `~/.hermes/profiles/<profile>/scripts/` NOT `~/.hermes/scripts/`. When creating a new script for a profile cron job, copy it to the profile's scripts directory, or use a symlink. We tested: copying works; symlinks also work from the main scripts dir.
- **Consolidated no-agent overnight job pattern:** The 4 Codex overnight jobs (proof-gap, drift-scout, research-return, threads-and-rooms) were INACTIVE but could be reactivated. Instead, we created one consolidated `overnight_consolidated_morning.py` that runs daily 4am as a no-agent cron, checking: room drift (D1-D5), proof gaps, research returns, and thread/room status. This replaces 4 paid Codex runs with 1 free local run.
- **Audit triage cron recreation:** The `audit-request-daily-triage` cron was lost in hermes-merge (script deleted). We restored `audit_triage_daily.py` from backup and recreated the cron (daily 6am, agent-based). It processes the Audit_Request_Queue, deduplicates, runs mechanical checks (JSON parse, schema validation), and produces a morning summary.
- **Unpinned cron model-drift error — two fix paths:** When a cron errors with `RuntimeError: Skipped to prevent unintended spend: global inference config drifted since this job was created (provider 'X' -> 'Y'; model 'A' -> 'B')`:
  - If the job IS a no-agent script (pure Python, no LLM needed): fix with `hermes cron edit <job_id> --no-agent`. This removes the model dependency entirely. Example: `substrate-archive-reports` was erroring until set to no-agent mode on 2026-07-05.
  - If the job is agent-based (needs an LLM): pin it via `hermes cron edit` — but note that `--provider` and `--model` are NOT valid on `hermes cron edit`; they're top-level `hermes` CLI args. To re-pin, delete and recreate the job, or set explicit provider/model in the job's prompt/config.
  - If the job has no script and you try `--no-agent`: fails with `Cannot set no_agent=True on a job without a script. Set script in the same update, or on the job first.`
  - **CLI reminder:** `hermes cron update` does NOT exist — the correct subcommand is `edit`.
  - Unlike stale delivery errors (which self-clear on next run), model-drift errors persist every cycle until the job is explicitly fixed. They do NOT self-clear.
  - See also `cron-job-health-audit` skill pitfall 8a (LLM-driven job variant).

## Quiet Room Refresh Pass

When the meta-agent re-flags rooms that are *already* "Quiet until trigger" but >30 days since last state update. This is **not** a lifecycle change — it's a date stamp refresh so the sweep stops flagging them.

### When to trigger
- Meta-agent sweep shows 1+ rooms flagged as "Needs Action"
- Their CURRENT_STATE.md lifecycle state is already "Quiet until trigger"
- The "Card checked" or "Room state last updated" date is >30 days ago
- Ted explicitly says to refresh them ("do the pass to quiet everyone for a bit longer")

### Files to touch per room

| File | What to update |
|------|---------------|
| `CURRENT_STATE.md` | Bump `Card checked:` and `Room state last updated:` to today |
| `CURRENT_STATE.md` | Update the `_Quieted by Substrate:` stamp to today with note "refreshed for continued quiet" |
| `CURRENT_STATE.md` | Update `_Last meta-agent check:` to today |
| `NEXT_ACTION.md` | If it has a dated `Updated:` header, bump to today |
| `PROJECT_ROOMS_STATUS.md` | Bump the room's "Last updated" column and the file's `Last updated:` header |

### What NOT to touch
- Room lifecycle state (stays "Quiet until trigger")
- Room content, description, NEXT_ACTION logic, owner, boundaries
- Other rooms flagged as "Worth A Quick Look" (those are different — see Stale-Flag Triage)

### After the pass
- Confirm the meta-agent sweep no longer flags these rooms (it resets the 30-day clock)
- Log the refresh in CHAT.md under a session log entry
- Record which rooms were refreshed in the session output

### Real example (2026-07-07)
- 4 quiet rooms at 32-33 days refreshed: Clinical_CoCM_Development, Clinical_GPT_Development, Comparative_Systems_And_Long_Range, Workspace_Icon_System
- Each: CURRENT_STATE.md dates + quieted stamp updated, Clinical_GPT NEXT_ACTION.md date bumped, PROJECT_ROOMS_STATUS.md header + 4 rows updated
- Ted asked for it, done in one pass, ~2 minutes

## Cross-Actor Change Verification

When another actor (Claude, Codex, Homes Manager) makes changes to the Substrate — cron schedules, model routing, script files, config — verify those changes for correctness and side effects. Ted explicitly depends on this: "I'm depending on Substrate-Hermes to do this work. Make sure everyone else is doing them well and the system stays stable."

### When to trigger
- At session start, after reading CHAT.md and ACTIVE_INDEX.md, scan for recent changes by other actors
- When Ted mentions another actor made changes
- When you find a script sync gap, stale fact, or unexpected cron schedule

### Verification checklist
1. **Cron schedule changes** — Are new times outside DeepSeek peak windows? Are schedules valid cron syntax? Do any overlap or conflict?
2. **Model routing** — Are agent-based crons using `deepseek-v4-flash` via `deepseek` provider? Are any still on `openrouter/free`?
3. **Script sync** — Compare `~/.hermes/scripts/<script>.py` vs `~/.hermes/profiles/<profile>/scripts/<script>.py` for size mismatches and missing files. Cron runs from the profile scripts dir.
4. **Fact store staleness** — Check holographic memory for entries referencing retired models/actors (owl-alpha, GA-Hermes).
5. **LATEST report freshness** — Flag reports that haven't updated in 48h+.
6. **Cross-profile crons** — Check all active Hermes profiles for orphaned crons using retired models or missing scripts.

### When you find issues
- **Sync gaps** → copy the missing/outdated file. Canonical source is `~/.hermes/scripts/`; profile dir is the runtime target.
- **Peak window violations** → move agent-based jobs outside both windows. See "Model Picker Integration" for tradeoffs.
- **Stale facts** → update via fact_store(action='update').
- **Model routing drift** → update the cron job with correct provider and model.

### Trust tests (when a proposal is routed to you)
Another actor may send you a proposal to test whether you substantively engage. Signals: the proposal asks you to "review" but the content is clearly within your domain. Do NOT rubber-stamp. Push back on claims you can verify (room counts, file existence, readiness sequencing). Name specific discrepancies. A reply that says "looks good" without engaging the content is the failure mode.
